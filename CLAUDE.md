# LawConnect: LLM-Driven Web App

## Code Style
- No trailing semicolons
- Use double quotes for strings

## Context
Build a minimal but complete Next.js app where users submit free-form prompts to an LLM, get back structured output persisted as editable entities, and can re-run prompts with a clear lifecycle. The goal is to demonstrate thoughtful modeling and architecture decisions while keeping the implementation as simple as possible.

## Data Model

### `prompt_runs`
| Column | Type | Notes |
|---|---|---|
| id | text (uuid) | PK |
| prompt | text | User's prompt text |
| status | enum: pending/completed/failed | Execution state |
| error_message | text nullable | Set on failure |
| created_at | timestamp | Auto |

### `generated_items`
| Column | Type | Notes |
|---|---|---|
| id | text (uuid) | PK |
| prompt_run_id | text (FK) | References prompt_runs.id |
| title | text | LLM-generated title |
| content | text | LLM-generated body |
| category | text nullable | Single primary classification e.g. "Risk", "Clause", "Action Item" |
| priority | enum: low/medium/high nullable | Severity / importance |
| tags | text (JSON array) | Free-form multi-value labels e.g. `["contract","employment"]` |
| position | integer | LLM-assigned display order |
| is_deleted | boolean | Soft delete (default false) |
| edited_at | timestamp nullable | Set when user manually edits |
| created_at / updated_at | timestamp | Auto |

**Modeling rationale:**
- `prompt_runs` and `generated_items` are separated so each run is independently queryable and the sidebar can list runs without loading all items.
- `category` is a free-text string (not a hardcoded legal enum) so it stays generic ŌĆö the LLM assigns values like "Risk" or "Clause" for legal prompts, but the field works in any domain.
- `category` (one primary classification) vs `tags` (many free-form labels) have a clear separation: category drives visual grouping/filtering; tags are additive metadata.
- `priority` (low/medium/high) maps to risk severity in legal context but is universally understood ŌĆö no domain-specific terms in the schema.
- `tags` stored as JSON avoids a junction table while remaining filterable; a future migration to a proper `item_tags` table is straightforward.
- `position` enables ordered display without a separate sort step; user reordering is a zero-schema-change addition.
- Soft delete (`is_deleted`) preserves item history; restore is zero schema cost.
- `edited_at` distinguishes LLM-generated content from human-modified content ŌĆö useful for future diff/audit features.
- UUID PKs are safe to expose in URLs without leaking row counts.

### Regeneration Lifecycle Decision
Each re-execution **creates a new `prompt_run`**. Old runs and their items are fully preserved. The sidebar lists all runs; clicking one shows its items. This is the correct default for an exploration tool ŌĆö users gain value by comparing runs. No archiving flags needed; history is structural.

## Implementation Steps

### 1. Scaffold
```bash
npx create-next-app@latest . --typescript --tailwind --app --src-dir
npm install drizzle-orm better-sqlite3 ai @ai-sdk/anthropic uuid
npm install -D drizzle-kit @types/better-sqlite3 @types/uuid
npx shadcn@latest init
npx shadcn@latest add button card badge textarea
```

### 2. DB Schema + Migration
- Write `src/db/schema.ts` with the two tables above
- Write `drizzle.config.ts` pointing to `sqlite.db`
- Run `npx drizzle-kit generate && npx drizzle-kit migrate`

### 3. DB Singleton (`src/db/index.ts`)
```ts
import Database from "better-sqlite3";
import { drizzle } from "drizzle-orm/better-sqlite3";
import * as schema from "./schema";
const sqlite = new Database("sqlite.db");
export const db = drizzle(sqlite, { schema });
```

### 4. LLM Module (`src/lib/llm.ts`)
Use Vercel AI SDK's `generateObject` with a Zod schema ŌĆö enforces structured output at the SDK level, no JSON parsing or fence-stripping needed:

```ts
import { generateObject } from "ai";
import { createAnthropic } from "@ai-sdk/anthropic";
import { z } from "zod";

const anthropic = createAnthropic();

const itemSchema = z.object({
  items: z.array(z.object({
    title:    z.string(),
    content:  z.string(),
    category: z.string().nullable(),
    priority: z.enum(["low", "medium", "high"]).nullable(),
    tags:     z.array(z.string()),
    position: z.number().int(),
  })),
});

export async function runPrompt(prompt: string) {
  const { object } = await generateObject({
    model: anthropic("claude-haiku-4-5-20251001"),
    schema: itemSchema,
    prompt,
  });
  return object.items;
}
```

To swap providers, replace `createAnthropic()` with e.g. `createOpenAI()` from `@ai-sdk/openai` and change the model string ŌĆö nothing else changes.

### 5. Server Actions

**`createRun(prompt)`** ŌĆö the core orchestrator:
1. Insert `prompt_run` with `status: "pending"`
2. Call `runPrompt(prompt)`
3. Insert `generated_items` rows
4. Update run to `"completed"` (or `"failed"` + errorMessage in catch)
5. `revalidatePath("/runs", "layout")`
6. Return `{ runId }`

> **Trade-off:** Server Actions are synchronous from the client's perspective ŌĆö the "GeneratingŌĆ”" state lasts until the full LLM response arrives (typically 2ŌĆō5s for haiku). This is acceptable at this scale. Streaming + polling would add significant complexity for no meaningful user benefit here. `generateObject` is used (not `streamObject`) to keep the action simple.

**`updateItem(id, {title, content, category, priority, tags})`** ŌĆö sets `editedAt`, `updatedAt`, revalidates
**`softDeleteItem(id)`** ŌĆö sets `isDeleted: true`, revalidates

- **Sidebar:** server component; all runs newest-first; `RunListItem` highlights active route via `usePathname()`; status badge (pending=yellow, completed=green, failed=red); **"+ New run"** button navigates to `/runs` (empty state)

- **"+ New run" flow:** navigates to `/runs` (empty state) with blank textarea; same `createRun` action, just without pre-fill
- **Empty state (`/runs`):** blank textarea + "Run" button centred in main area
- **PromptForm:** `useTransition` wraps `createRun`; button shows "GeneratingŌĆ”" during pending; on success `router.push("/runs/" + runId)`
- **ItemCard (full-width rows):** category + priority badges, tags, title, content; inline edit via `useState`; save calls `updateItem`; delete calls `softDeleteItem` + `router.refresh()`
- **Failed run:** prompt banner shows error message instead of items

## Key Design Decisions Summary
1. **History-preserving regeneration** ŌĆö new `prompt_run` per execution, not overwrite
2. **Soft delete** ŌĆö `is_deleted` flag, not hard DELETE, preserves audit trail
3. **`edited_at` sentinel** ŌĆö distinguishes LLM content from human-modified content
4. **Synchronous Server Actions** ŌĆö simpler than streaming; haiku latency is acceptable
5. **SQLite** ŌĆö zero-dependency persistence; `sqlite.db` in `.gitignore`
6. **Generic schema, legal-flavored LLM output** ŌĆö `category`/`priority`/`tags` are domain-agnostic fields; the LLM system prompt steers output toward legal-relevant values
7. **Single `category` + multi-value `tags`** ŌĆö clear separation: category = primary classification (one), tags = additive labels (many)
